## 1. Vision & Positionierung

### 1.1 Elevator Pitch

**Canzoia — eine deutsche, familien-zentrierte Audio-App — wie Spotify, aber die Inhalte werden live und persoenlich fuer jeden in deiner Familie von AI generiert.**

Jedes Familienmitglied hat ein eigenes Profil. Aus einem Katalog von "Shows" (AI-Charakteren mit Domain-Wissen) generiert die App on-demand personalisierte Audio-Episoden: Gutenacht-Geschichten fuer die Kinder, Meditations-Sessions fuer die Eltern, Wissens-Podcasts fuer Senioren. Die UX ist die von Apple Podcasts oder Spotify — vertraut, poliert, unkompliziert. Die Technologie darunter generiert alles frisch.

### 1.1.1 Brand: Canzoia

**Etymologie:** Kunstwort aus italienisch *canzone* (Lied) + *gioia* (Freude). Phonetisch weich, dreisilbig, einpraegsam — fuehrt die warme-mediterrane Stimmung, die die Plattform vermitteln soll, schon im Namen.

**Aussprache:**
- DE: **Kan-TSOI-a**
- IT: Kan-TSO-ja
- EN: Can-ZOY-a

**Schreibweise:** Unmissverstaendlich — kein Doppel-Buchstabe, kein Umlaut, keine Apostrophe. Gleich beim ersten Hoeren erkennbar.

**Markenkern:**
- Warmth (vs. Tech-cold)
- Family (vs. Solo-Consumer)
- Personal (vs. Catalog)
- Songs & Stories (vs. Podcasts)

### 1.1.2 Tagline-Kandidaten (noch zu waehlen)

**DE — Primary Market:**
1. *"Jede Geschichte fuer dich geschrieben."* ← narrativ, direkt
2. *"Fuer jeden in deiner Familie. Jeden Tag neu."* ← Family-Multi-Profile-Pitch
3. *"Hoerbar persoenlich."* ← kurz, Claim-fertig
4. *"Geschichten, die dich kennen."* ← Profile-Learning als Hook
5. *"Die Audio-Bibliothek, die fuer dich schreibt."* ← deskriptiv
6. *"Lieder, Geschichten, Wissen — fuer jeden Ton deiner Familie."* ← Wortspiel Ton=Audio+Stimmung

**EN — Secondary (App-Store Screenshots, internationale Pitches):**
1. *"Every story, made for you."*
2. *"The audio library that knows you."*
3. *"Personal audio. For everyone."*

**Empfehlung (Tom entscheidet):** #3 *"Hoerbar persoenlich."* als primaerer Claim — kurz, Deutsch-idiomatisch, funktioniert als Logo-Subline UND App-Store-Tagline. Langform fuer Marketing-Copy: *"Jede Geschichte fuer dich geschrieben — fuer jeden in deiner Familie."*

### 1.2 Was es NICHT ist

- **Kein "AI-Podcast-Creator-Tool"** (die Kategorie ist verbrannt durch Jellypod, Podcastle, Wondercraft).
- **Kein "AI-Companion"** (zu ueberlaufen: Replika, Character.ai, Pi).
- **Kein "Hoerbuch-Katalog"** (Audible / Storytel / BookBeat besetzen das).
- **Keine medizinische Beratung** — Health-Inhalte nur als Research-Assistent ("finde neue Studien zu meinem Profil"), nie als Rat.

## 3. Ziel-User & Profile

### 3.1 Primaer (Launch): Familien mit Kindern 3–10

- Eltern (30–45), kaufkraeftig, digital-affin
- Nutzen heute Tonies, Tigertones, Audible
- Schmerzpunkte: Inhalte werden langweilig, Kinder wollen immer "das Gleiche nochmal", keine personalisierten Lern-Inhalte moeglich, kein Zugriff auf neue Themen ohne Kauf
- Bereit fuer €9.99–14.99/Monat wenn der Wert klar ist

### 3.2 Sekundaer (Phase 2): Erwachsene & Senioren in gleicher Family

- Meditation / Wellness / Schlaf-Geschichten
- Lernen (Sprachen, Hobbys, Wissen)
- Gesundheits-Research (NICHT Beratung)

### 3.3 Profil-Modell

```
FamilyAccount
├── Profile[] (1..6)
│   ├── id
│   ├── name (z.B. "Lena, 7 Jahre")
│   ├── type: "kid" | "adult" | "senior"
│   ├── attributes: { key: value } — Freitext-Datenbank
│   │   z.B. { "interesse": "Dinosaurier", "lieblingstier": "Koala",
│   │          "sprachlevel": "Grundschule 2. Klasse",
│   │          "geht_heute_zu": "Oma" }
│   ├── listeningHistory[]
│   └── preferences { sleepTimer, voiceSpeed, volume }
```

**Prinzip: Profil wird passiv aufgebaut.**
- Erstes Onboarding: 3-Min-Chat mit Host ("Stell dich mir vor — was magst du gerade?")
- Waehrend der Nutzung: Host fragt periodisch Follow-Ups ("Darf ich mir merken, dass du heute bei Oma bist?")
- NIEMALS ein leeres Formular mit 20 Feldern.

### 3.4 Profile-Learning — die App lernt den User dauerhaft

**Kernprinzip:** Je laenger ein User die App nutzt, desto besser werden die Ergebnisse. Das ist der eigentliche Wettbewerbsvorteil gegenueber generischen AI-Story-Apps.

**Was gespeichert wird:**

```
Profile
├── explicitAttributes     // User hat selbst gesagt / bestaetigt
│   └── { "alter": 7, "lieblingstier": "Koala", ... }
├── implicitAttributes     // AI hat aus Verhalten abgeleitet
│   └── { "bevorzugteHoerdauer": "15 Min", "hoertMeistAm": "Abend",
│          "reagiertStarkAuf": ["Tiere", "Abenteuer"],
│          "mag_offenbar_NICHT": ["laute Szenen"] }
├── interactionLog[]       // Unveraenderlicher Audit-Trail
│   └── { episodeId, startedAt, listenedSeconds, completed,
│          rating, userQuote?, skippedSegments? }
└── preferences            // Direkt editierbar im UI
    └── { sleepTimer, voiceSpeed, volume, nightMode }
```

**Learning-Zyklen:**

1. **Waehrend der Episode** (Live-Signal):
   - Wie lange gehoert? Abgebrochen bei welchem Segment?
   - User pausiert/spult = schwaches Negativ-Signal

2. **Nach der Episode** (Optional 5-Sek-Feedback):
   - 3-Emoji-Rating: Liebe / OK / Mag-nicht
   - Freitext-Option: "Was sollen wir anders machen?"

3. **Passive Follow-Ups im Flow** (Host fragt):
   - "Du hast Dinosaurier erwaehnt — darf ich mir merken, dass du die magst?"
   - "Gerade eine Herbst-Geschichte gefeiert — magst du auch Winter-Themen?"

4. **Woechentlicher Background-Polish**:
   - Inngest-Job aggregiert InteractionLog in `implicitAttributes`
   - Beispiel: "5 von 5 letzten Episoden nach 20 Uhr gestartet → `bevorzugteUhrzeit: abends`"
   - Attribute werden mit `confidence` gespeichert (0-1), mit jedem Signal staerker/schwaecher

5. **Profile-Dashboard** (DSGVO-konformer Transparenz-Layer):
   - User sieht ALLES was die App ueber ihn weiss
   - Kann jedes Attribut editieren oder loeschen
   - "Export mein Profil" als JSON (DSGVO Art. 20)
   - "Loesche alles" (DSGVO Art. 17) — kaskadiert auf InteractionLog

**Prompt-Integration:**
Bei jeder Episoden-Generierung wird das komplette Profil-JSON in den LLM-System-Prompt injiziert. Je reicher das Profil, desto persoenlicher das Ergebnis. Das ist der Grund warum sich Nutzung lohnt — der Wert steigt ueber Zeit.

### 3.5 Background-Generierung — User wartet nicht

**Pflicht-Feature:** Jede AI-Generierung laeuft im Hintergrund. User klickt "Generiere Episode" → UI zeigt "Wird erstellt..." → User kann die App schliessen, ins Bett gehen, auf eine andere Seite navigieren. Kein Abbruch, keine verlorene Arbeit.

**Implementation:**
```
User klickt "Generiere Episode" in Show X
    ↓
POST /api/episodes/generate
    ↓
1. Validierung (Budget ausreichend? Profil vorhanden?)
2. Budget-Abzug (sofort, wird bei Fehler refunded)
3. Inngest-Job triggern mit { profileId, showId, formatId }
4. Response: { jobId, status: "queued" }
    ↓
UI zeigt "Wird erstellt..." + optimistischer Library-Eintrag
User kann App schliessen
    ↓
Inngest-Worker (separater Execution-Context):
5. LLM-Call (GPT-4o-mini) — Skript generieren
6. Azure TTS — Audio synthetisieren (parallel pro Segment)
7. FFmpeg-Merge im Worker
8. Upload zu Cloudflare R2
9. DB-Update: Episode.status = "ready"
10. Supabase Realtime → UI bekommt Update (falls offen)
11. Push-Notification via FCM/APNs → Phone klingelt
    ↓
User tippt auf Push → App oeffnet direkt in Player
```

**Error-Handling:**
- Auto-Retry 3x bei TTS-Fehler
- Bei finalem Fehler: Budget refund + Error-Push ("Episode konnte nicht erstellt werden")
- Alle Fehler in Sentry + Inngest-Dashboard sichtbar

**Wichtiger UX-Nebeneffekt:** User kann parallel 3 Episoden auf die Pipeline legen, dann einfach warten. Das macht die App "aktiv" auch wenn der User gerade nicht hoert.

## 5. Business Model & Pricing

### 5.1 Tiers — Minuten-Budget-Modell

Statt Credit-Mathe oder starrer Episoden-Zaehlung arbeiten wir mit einem **Minuten-Budget pro Monat**. User sieht transparent: "Du hast 200 Min. Diese Episode kostet 15 Min." Kein Lernaufwand, maximale Klarheit.

| Tier | Preis | Budget/Monat | Profile | TTS-Qualitaet |
|---|---|---|---|---|
| Free | €0 | 30 Min (non-rollover) | 1 | Azure Basic |
| **Family** | **€9.99/Mo** oder **€89/Jahr** (25% off) | 200 Min | 6 | Azure Neural |
| Family Plus | €14.99/Mo oder €139/Jahr | 500 Min | 6 | Azure + ElevenLabs Flagship |
| Top-up | €2.99 | +50 Min | — | Aktueller Tier |
| **Lifetime Early-Adopter** | **€199** (erste 1.000 User, zeitlich limitiert) | Family lebenslang | — | — |

### 5.2 Context-Pricing — was eine Episode intern kostet

Verschiedene Inhalte brauchen unterschiedlich viel Budget:

| Content-Typ | Budget-Kosten | Begruendung |
|---|---|---|
| Standard-Episode (Azure TTS) | **1× Laenge** | 10-Min Episode = 10 Min Budget |
| Premium-Show (ElevenLabs Flagship) | **1.5× Laenge** | 10 Min = 15 Min Budget |
| Deep-Dive-Research-Episode | **1× Laenge** | Wie Standard |
| AI-Research-Agent (Text-Output) | **0.5× Laenge** | Kein TTS noetig, nur LLM |
| Regenerierung einer existierenden Episode | **0.5× Laenge** | Budget-Discount fuer "nochmal mit Anpassung" |

So kann der User ausprobieren ohne Angst, ein Premium-Format bekommt aber fair abgebildet mehr Budget.

### 5.3 Warum dieser Mix

- **Freemium:** 30 Min/Monat = 2–3 Episoden um den Wert zu erleben, reicht nicht fuer Nutzung. Conversion-Benchmark: 10–25% (Replika liegt bei 25%).
- **Jahres-Rabatt:** DACH bevorzugt Monats-Preise, aber 25% Annual-Discount konvertiert Vielnutzer.
- **Lifetime:** Cash-Bootstrap. 500 verkaufte Lifetimes = €100k Vorab-Revenue = 12 Monate Runway.
- **Top-ups:** Power-User ohne Tier-Wechsel. 50-Min-Pakete sind psychologisch kleiner Commit als "noch ein Abo hochstufen".

### 5.3 Unit-Economics-Rechnung

**Family-Tier, 20 Episoden/Monat (je 10 Min, Azure Neural):**

**Family Plus, 50 Episoden, 10% ElevenLabs-Premium:**

| Posten | Kosten pro User/Monat |
|---|---|
| Azure (45) | $6.30 |
| ElevenLabs (5 × $0.90) | $4.50 |
| LLM + Infra | $1.20 |

→ Family-Tier ist die Cash-Cow. Family Plus ist Premium-Positioning mit duennerer Marge.

## 7. Content-Strategie — wie Shows gebaut werden

### 7.1 Show-Definition

Eine Show ist ein JSON-Objekt mit:

```ts
type Show = {
  id: string
  name: string                    // "KoalaTree Familie"
  tagline: string                 // "Geschichten fuer die ganze Familie"
  coverImageUrl: string           // Via Engine generiert
  trailerVideoUrl?: string        // Fuer TikTok/Instagram
  hosts: Host[]                   // 1..N Actors aus Engine
  formats: Format[]               // "Gutenacht-Geschichte", "Lern-Geschichte"
  targetProfiles: "kid" | "adult" | "senior" | "any"
  promptSystem: string            // System-Prompt der Show-Persoenlichkeit
  contentPolicy: ContentPolicy    // Was darf/darf nicht generiert werden
}

type Format = {
  id: string                      // "gutenacht-10min"
  name: string                    // "Gutenacht-Geschichte"
  lengthMinutes: number           // 10
  structure: string               // Prompt-Template
  creditCost: number              // 1 / 2 / 3
}
```

### 7.2 Episoden-Generierung (Pipeline)

```
User klickt "Neue Episode" in Show X, Format Y
    ↓
1. Lade Profile-Kontext (name, alter, interessen, last-episodes)
    ↓
2. Lade Show-System-Prompt + Format-Template
    ↓
3. Merge in LLM-Call (GPT-4o-mini):
   "Du bist Koda aus KoalaTree Familie. Generiere eine 10-Min
    Gutenacht-Geschichte fuer Lena (7 Jahre, mag Dinosaurier,
    geht morgen zur Oma). Format: [structure template]"
    ↓
4. LLM liefert strukturiertes Skript:
   {
     "title": "Der Dino im Koala-Wald",
     "segments": [
       { "voice": "Koda", "text": "..." },
       { "voice": "Kiki", "text": "..." }
     ]
   }
    ↓
5. Fuer jeden Segment: Azure TTS Call mit Voice-ID
    ↓
6. Audio-Files zusammenfuegen (ffmpeg oder Web-Audio)
    ↓
7. Upload zu Vercel Blob
    ↓
8. DB-Entry: Episode gespeichert, Credits abgezogen
    ↓
9. Push an User: "Deine Episode ist fertig"
```

**Dauer pro Episode:** ~30–60 Sek (LLM 10s, TTS 20–40s parallel, Merge 5s, Upload 5s). User kann parallel browsen.

### 7.3 Show-Assets via KoalaTree-Engine

- **Cover-Art:** GPT-Image-1.5 mit Show-Style-Prompt. Einmalig pro Show.
- **Trailer-Video:** 15-Sek-Clip mit Hosts, fuer TikTok/Instagram/Reels. Multi-Variante (3–5 pro Show).
- **Episoden-Cover (optional):** Thumbnail pro Episode, wenn User "share" klickt.

Die Engine bleibt ein interner Service, wird nur ueber typed API-Client angesprochen.

## 8. Compliance

### 8.1 EU AI Act

- Status: Eure App ist **Limited Risk** — nur Transparenz-Pflicht (User muss wissen dass AI im Spiel ist).
- Implementierung: "AI-generated"-Badge sichtbar in UI, FAQ-Eintrag, Footer-Disclaimer.
- **Deadline relevant:** Februar 2026 (Guidelines), August 2026 (High-Risk). Wir sind nicht High-Risk, also ueberwiegend safe.

### 8.2 DSGVO (besonders bei Kinderprofilen)

- **Art. 8 DSGVO:** Kinder unter 16 brauchen elterliche Einwilligung (DE-Standard).
- **Parental-Consent-Flow:**
  - Beim Anlegen eines Kid-Profils: Eltern-Email bestaetigt explizit per Click-Link.
  - Profiling-Einwilligung separat (checkbox, nicht pre-ticked).
- **Datenminimierung:** Nur Profil-Attribute die User selbst angegeben hat, keine implizite Tracking-Inferenz.
- **Recht auf Loeschung:** Ein-Klick-Profil-Loeschung, inkl. aller Audio-Files und Logs.
- **Datenstandort:** Supabase EU-Region oder Neon EU-Region. Azure TTS in West Europe.

### 8.3 Wellness vs. Medical

- **Wellness-Shows-Disclaimer** (unmissverstaendlich):
  > "Diese Inhalte dienen Wellness-Zwecken und ersetzen keine medizinische Beratung. Bei gesundheitlichen Beschwerden wende dich bitte an einen Arzt."
- **Research-Agent** (wenn spaeter gebaut):
  - Zeigt nur Studien-Zusammenfassungen + Quellen-Links
  - Keine "Was du tun sollst"-Empfehlungen
  - Claude als Research-Assistent, nicht Therapeut

### 8.4 Content-Moderation

- OpenAI Moderation API vor jedem TTS-Call
- Eigene Regel-Liste fuer Kids (Gewalt, Angst, Schlaf-Trigger)
- Incident-Flow: User meldet Episode → sofort gesperrt → manuelle Review → Prompt-Regel-Update
- **Phase 2:** Kinderpsychologen-Advisory-Board (1–2 Experten, bezahlt pro Review-Stunde)

## 11. Offene Entscheidungen

Noch nicht entschieden, muss vor bzw. bei Phase 0 geklaert werden:

1. **Tagline-Auswahl** — siehe Kap. 1.1.2 fuer Kandidaten. Empfehlung: *"Hoerbar persoenlich."*
2. **Brand-Architektur** — "Canzoia" Plattform-Name + "powered by KoalaTree" Badge? Oder komplett getrennt? Aktuelle Neigung: getrennt — Canzoia als Produkt-Brand, KoalaTree als Show-Brand in der App.
3. **Launch-Hauptsprache** — Deutsch-only zum Launch, oder Deutsch + Englisch parallel? Meine Empfehlung: Deutsch-only, niedrigere Kompetition, bessere UA-Kosten.
4. **Hosting** — Vercel (easy) vs. eigener EU-Server (DSGVO-sauberer). Neigung: Vercel EU-Region zum Launch, Migration bei Bedarf.
5. **Schriftart / Design-System** — Modern (Inter, Satoshi) oder warm (Recoleta, Fraunces)? Audio-App → warm & cozy, ich tendiere Recoleta oder Inter + warme Farben. Italian-Brand-Name spricht eher fuer eine serif wie Fraunces oder PP Editorial New.
